## 1. 설계 결정 사항 (반드시 준수)

### 1-1. IAM 정책은 2단계로 운영한다
- **Phase 1**: 개발용 느슨한 정책(리소스 ARN 한정 + 서비스 단위 액션 허용).
- **Phase 5**: 최소 권한 정책으로 재작성, 와일드카드 금지.

### 1-2. VPC는 "필요 시 도입"으로 전환한다
- 오케스트레이터/수집기 Lambda 모두 VPC 밖에서 시작.

### 1-3. Bedrock Knowledge Base는 데이터 준비 후 생성한다
- KB는 `support_target`/`support_content`/`overview`(API 제공 텍스트)를 임베딩해 의미검색 전용으로 사용하며, 정확한 수치 조회 용도로 쓰지 않는다. 정형 수치는 마스터 테이블(1-13)에 저장한다.

### 1-4. 초안 생성(`generate_draft`)은 "수치는 삽입, 서술은 생성" 원칙을 따른다
- 마스터 테이블의 지원대상/지역/금액/마감일/**신청링크/문의처**(v2.3 추가)는 LLM이 재생성하지 않고 그대로 삽입한다.
- LLM은 사업 개요, 신청 사유, 기대효과 서술 등 비정형 문장 생성에만 사용한다.

### 1-5. 기대효과 계산(`calc_expected_effect`)은 규칙기반 + LLM 설명 하이브리드로 구현한다

```
기대 매출증가(원)   = 지원금액 × 업종별 평균 매출승수(더미 테이블)
기대 고용창출(명)   = 지원금액 / 업종별 평균 1인당 투자비용(더미 테이블)
예상 순이익(원/년)  = 기대 매출증가 × 이익률 가정치        ← v2.3 신설
회수기간(년)        = 지원금액 / 예상 순이익
```

**v2.3 명시 — 계산식의 수학적 성질**: 회수기간 식을 전개하면 `지원금액 / (지원금액 × 매출승수 × 이익률) = 1 / (매출승수 × 이익률)`이 되어 지원금액이 소거된다. 즉 **회수기간은 지원금액 규모와 무관하게 업종별 매출승수·이익률에 의해서만 결정되는 비율 지표**이며, 이는 버그가 아니라 이 선형 모델의 수학적 성질이다. 지원금액 규모 차이를 사용자에게 보여주기 위해 **예상 순이익(절대값, 원 단위)** 지표를 함께 제시한다. 프론트(Phase 6)는 회수기간(비율)과 예상 순이익(절대값)을 함께 표시해야 한다.

**입력 계약**: `program_id`를 받아 마스터 테이블에서 지원금액을 조회한다 (`min_support_amount`/`max_support_amount` 우선, 없으면 `overview`/`support_content` 텍스트에서 LLM 보완 추출). 공고정보 API 수집 공고는 지원금액 필드가 대부분 LLM 추출값이므로, 추출 실패 시 기대효과 계산을 스킵하고 안내 메시지를 반환한다.

### 1-6. 모니터링 최소 구성은 Phase 2부터 조기 도입한다

### 1-7. 매칭 로직은 하드필터 + AI 랭킹의 2단계로 구성한다

- **1단계 (규칙 기반)**: 마스터 테이블 대상으로 아래 필드로 하드필터.
  - **`induty`(업종) 일치 여부 (v2.3 복원)**: 공고의 `induty`가 "전체/공통"이거나 사용자 입력 업종과 텍스트 키워드 매칭. API 공고는 hashTags/description에서 추출한 텍스트이므로 완화된(부분 포함) 매칭을 적용한다 (부록 A).
  - `area_code`에 `4700`(경상북도) 또는 `1000`(전국) 포함. API 공고는 hashTags에서 추출
  - `employee_count_code` 근로자수 구간 요건 충족. **API 공고는 빈값(미제공)이면 자동 통과**
  - `business_age_code` 업력 구간 요건 충족 (예비창업 대상 공고는 `fntnYn` 별도 확인). **API 공고는 빈값(미제공)이면 자동 통과**
  - `deadline >= 오늘`. API 공고는 `reqstDt`에서 파싱

  **다중값 파싱 규칙**: MANUAL(폴백) 데이터의 `employee_count_code`, `business_age_code`, `need_certification_code`, `sales_amount_code`는 `|` 구분 다중값이 가능하므로, 하드필터/가산조건 비교 전 반드시 해당 필드를 `|`로 split한 리스트로 변환한 뒤 각 값에 대해 포함 여부를 확인한다. API 수집 공고는 이 필드들이 빈값이므로 해당 필터를 자동 통과(빈값 = 조건 없음으로 간주)한다.

- **2단계 (AI 기반)**: 1단계 통과 후보만 KB 의미검색으로 희망 지원분야 관련도 순위화 + 보유 인증(`need_certification_code` 일치) 및 매출액 구간(`sales_amount_code` 일치, 입력된 경우만) 시 가산(필수 아님, API 공고는 빈값이므로 가산 없음) + LLM 매칭 근거 문장 생성. **근거 생성 프롬프트에는 코드값이 아닌 텍스트 필드(`area_name`, `induty`, `biz_type_name`, `support_type_name` 등)를 전달한다** (v2.3 — 코드값이 그대로 노출되는 것을 방지).
- 후보 0건 시 어떤 필터(업종/지역) 때문인지 안내 + 조건 완화 재검색 제안.

### 1-14. KB 의미검색은 메타데이터 필터링으로 후보를 실제로 제한한다 (v2.4 신설)

`search_programs`의 2단계 설명("1단계 통과 후보만 대상으로 KB 의미검색")은 Bedrock Knowledge Base의 Retrieve API가 기본적으로 **전체 인덱스**를 대상으로 검색한다는 사실과 맞부딪힌다. 이를 구현 없이 방치하면 개발자가 "KB에서 상위 N개를 받아온 뒤 애플리케이션에서 1단계 후보와 교집합을 구하는" 방식으로 임시 구현하기 쉬운데, 이 경우 **KB가 반환한 상위 N개가 전부 1단계 탈락 후보이면 교집합이 0건이 되는 위험**이 있다 (특히 후보 풀이 작은 해커톤 데이터셋에서 발생 확률이 높다).

**확정 방식**: Bedrock KB 인덱싱 시 각 공고 문서에 `program_id`, `area_code`, `employee_count_code`, `business_age_code`, `induty` 등을 **메타데이터**로 함께 색인한다 (S3 데이터소스 사용 시 `<원본파일명>.metadata.json` 사이드카 파일로 첨부). `search_programs`는 Retrieve API 호출 시 `vectorSearchConfiguration.filter`에 `{"in": {"key": "program_id", "value": [1단계 통과 후보 program_id 리스트]}}` 조건을 지정하여, **검색 자체를 1단계 통과 후보로 실제로 제한**한다. 이렇게 하면 교집합 0건 위험이 구조적으로 사라진다.

후보 리스트가 비정상적으로 커지는 경우(예: 전국 대상 공고가 다수 통과)를 대비해, filter에 넘기는 program_id는 상위 200개 등으로 상한을 두는 것을 권장한다 (부록 A).

### 1-15. 외부 동기 호출 체인은 API Gateway 타임아웃 예산 내에 들어오도록 설계한다 (v2.4 신설)

API Gateway(REST/HTTP API 공통)의 통합 타임아웃은 **최대 29초**다. `POST /profile`은 `verify_certifications`(증명서 API 최대 3회 호출 + 429 재시도 대기) → `search_programs`(하드필터 + KB 검색 + Nova Pro 근거생성 LLM 호출)를 순차로 거치는데, 재시도 대기시간과 LLM 호출 지연이 겹치면 29초를 넘겨 사용자에게는 정상 처리 중인데도 504 에러가 표시될 수 있다.

**확정 방식**: `verify_certifications`의 429 재시도는 **최대 1회**로 제한하고, 응답의 `retryAfterSeconds`가 5초를 초과하면 재시도를 포기하고 "인증정보 없음"으로 처리한 뒤 즉시 다음 단계(`search_programs`)로 진행한다. 인증서 확인은 이후 마이페이지에서 수동 재조회할 수 있는 여지를 남긴다. 이 규칙으로 `/profile` 체인의 최악 소요시간 상한을 예측 가능한 범위로 관리한다.

### 1-16. 외부 API(공고/증명서) 호출 공통 원칙 (v2.4 신설)

- **재시도**: 증명서 API의 429뿐 아니라, 공고정보 API(bizinfo.go.kr) 호출도 네트워크 오류·일시 장애에 대비해 지수 백오프 재시도(최대 2회)를 공통 유틸로 적용한다. 단, 1-15 원칙에 따라 동기 호출 체인(`/profile`)에서는 재시도로 인한 총 대기시간 상한을 지킨다.
- **민감정보 로그 마스킹**: 공고정보 API의 `crtfcKey`(query parameter로 전달되어 요청 URL에 노출됨)와 증명서 API의 `bizno`(사업자번호)는 CloudWatch 로그에 평문으로 남기지 않는다. Lambda의 요청 로깅 시 해당 값을 마스킹(예: 뒤 4자리만 표시)하거나 로그에서 제외한다.
- **Secrets Manager 캐싱**: 공고정보 API의 `crtfcKey`와 증명서 API의 `token`은 Lambda 호출마다 Secrets Manager를 조회하지 않고, 실행 환경의 전역 변수에 TTL(예: 1시간)을 두어 캐싱한다. 콜드스타트 시에만 조회하여 지연시간과 Secrets Manager 호출 비용을 줄인다.

### 1-17. Guardrails 적용은 Bedrock Agent 없이도 가능하다 (v2.4, 1-11 보완)

`generate_draft`는 Bedrock Agent를 거치지 않고 Lambda가 Nova Pro를 직접 호출하는 구조(1-8)이므로, Guardrails 역시 Agent 연결이 아니라 **Converse API(또는 InvokeModel) 호출 시 `guardrailConfig` 파라미터로 Guardrail ID/버전을 직접 지정**하는 방식으로 적용한다. Bedrock Agent를 쓰지 않는다고 해서 Guardrails를 못 붙이는 것이 아님을 팀 전체가 인지해야 한다 (1-11의 grounding source 지정과 함께 적용).

## Phase 4 — 백엔드 오케스트레이션 (Lambda: Strands Agent)

| 번호 | 도구 | 트리거 방식 | 분류 | 내용 |
|---|---|---|---|---|
| 1 | `search_programs` | 프로필 제출 시 자동 | 직접 함수 호출 | 1-7 원칙: (a) 업종/지역/근로자수/업력/마감일 하드필터(다중값 파싱 포함) → (b) **Bedrock Retrieve API를 `program_id` 메타데이터 필터와 함께 호출**해 1단계 후보 내에서만 KB 순위화(인증·매출액 가산, 1-14) → (c) Nova Pro로 매칭 근거 생성(텍스트 필드 사용) → (d) 0건 시 조건완화 제안 |
| 2 | `verify_certifications` | 사업자번호 입력 시 자동 | 직접 함수 호출 | 증명서 API(y104/y105/y106) 호출. **429 시 최대 1회, `retryAfterSeconds` 5초 이하일 때만 재시도(1-15)**, 초과 시 인증정보 없음으로 즉시 진행 |
| 3 | `generate_draft` | "초안 생성" 클릭 시 자동 | 직접 함수 호출 | 마스터 테이블 조회 후 템플릿 삽입(수치+**신청링크**), Nova Pro 서술 생성, Guardrails 적용 |
| 4 | `calc_expected_effect` | 초안 생성 시 자동 | 직접 함수 호출 | `program_id`로 금액 조회 → 매출증가/고용창출/**예상순이익**/회수기간 계산 → Nova Pro 설명 |
| 5 | `get_matched_programs` | 챗봇 질의 | Agent 도구 선택 | 이전 매칭 재조회 |
| 6 | `explain_program` | 챗봇 질의 | Agent 도구 선택 | 공고 세부 설명 |
| 7 | `revise_draft` | 챗봇 질의 | Agent 도구 선택 | 초안 수정 |

**완료 기준**: `search_programs`가 제조·전자 외 업종 더미 데이터를 하드필터에서 제외함을 확인. `calc_expected_effect`가 예상순이익(절대값)과 회수기간(비율) 두 지표를 모두 반환함을 확인. **`/profile` 체인(`verify_certifications`+`search_programs`)의 최악 케이스(429 1회 재시도 포함) 소요시간이 API Gateway 29초 제한 내에 들어오는지 부하 없는 환경에서 측정(1-15)**.

## 부록 C. 실행 순서 권장안 (Phase 번호 ≠ 착수 순서)

Phase 1~8 번호는 **설계상 의존관계**를 보여주기 위한 것이며, 팀원들이 반드시 이 순서대로 "하나씩 끝내고 다음으로" 넘어가야 한다는 뜻이 아니다. 실제로는 아래처럼 병렬로 진행해야 정해진 시간(Day1 15:00 개발 시작 ~ Day2 16:00 개발 완료) 안에 끝낼 수 있다.

### 병렬 트랙
- **트랙 A (인프라/데이터)**: Phase 1(보안기반, 공고정보 API `crtfcKey`는 발급 완료이므로 즉시 Secrets Manager에 등록. 증명서 API는 승인 대기 중) → Phase 2(테이블/버킷) → Phase 3(API 연동). 증명서 API 승인이 늦어져도 Phase 3의 공고 수집은 즉시 진행 가능. Phase 2에서 확보한 폴백 5건(`MANUAL-001`~`005`)도 병행하여 트랙 B/C에 데이터를 즉시 공급한다.
- **트랙 B (백엔드 로직)**: 트랙 A의 폴백 5건이 마스터 테이블에 올라가는 즉시, API 연동 완료 여부와 무관하게 Phase 4(7개 도구)를 착수한다. 실 API 연동은 나중에 붙여도 로직 자체는 동일하다.
- **트랙 C (프론트엔드)**: Phase 5의 API 요청/응답 JSON 스펙을 **Day1 개발 착수 직후 가장 먼저 팀 전체가 확정**하고, 그 스펙에 맞는 목업(mock) 응답으로 화면부터 만든다. 백엔드가 실제로 완성되기 전까지 목업으로 개발하다가, 완성되는 대로 실제 엔드포인트로 교체한다.
- **트랙 D (인증/엣지)**: Cognito 앱 클라이언트는 임시 콜백 URL로 먼저 만들어두고, CloudFront 배포가 나온 뒤 콜백 URL을 갱신한다 (Phase 5 표 참고). CORS는 API Gateway가 만들어지는 시점에 바로 설정해, 프론트-백엔드 연결 시점에 디버깅 시간을 낭비하지 않는다.

### Day1 안에 반드시 끝내야 할 것 (이후 일정이 순연되면 여기까지는 사수)
1. 공고정보 API `crtfcKey` Secrets Manager 등록 (✅ 발급 완료) + 증명서 API 승인 확인
2. Bedrock 모델 액세스 활성화 확인
3. 폴백 5건 확보 및 마스터 테이블 적재 (KB 메타데이터 색인 포함, 1-14)
4. `/profile`, `/draft`, `/chat`의 요청/응답 JSON 스펙 확정 (프론트-백엔드 계약)

### 발표 시연 리스크 관리
라이브 API 데이터는 실시간으로 바뀔 수 있어 발표 당일 매칭 결과가 리허설과 달라질 위험이 있다. **발표에 사용할 회사 프로필 시나리오를 최소 1개, 폴백 5건 중 확실히 매칭되는 조합으로 고정**해 리허설하고, 라이브 API 데이터가 예상과 다르게 나오더라도 이 고정 시나리오로 즉시 전환할 수 있게 준비한다.
