## A. 권한 없지만 우회하여 구현 성공한 항목

이 항목들은 스펙에 명시된 방식과 다르게 구현했지만, **기능적으로 동등한 결과**를 달성했습니다. 실 서비스 전환 시 원래 방식으로 교체하면 되며, 현 상태에서도 데모/발표에는 문제 없습니다.

| # | 스펙 요구사항 | 부족한 권한 | 우회 구현 | 전환 난이도 |
|---|---|---|---|---|
| 1 | Secrets Manager에 crtfcKey/token 저장 + TTL 캐싱 (1-16) | `secretsmanager:*` | Lambda 환경변수로 저장. 코드에 `os.environ.get()` 사용 | 낮음 — SM 시크릿 생성 후 코드 3줄 교체 |
| 2 | KMS 고객관리형 키로 암호화 | `kms:*` | AWS 관리형 키(aws/dynamodb, aws/s3) 자동 적용 | 낮음 — CMK 생성 후 테이블/버킷에 지정 |
| 3 | Bedrock KB + Retrieve API filter (1-14) | `iam:PassRole` (to bedrock.amazonaws.com), `aoss:*` | LLM 리랭킹으로 대체. Nova Lite가 후보 20건을 적합도순 재정렬 | 중간 — KB 생성→DataSource 연결→orchestrator 코드 교체 |
| 4 | EventBridge 1일 1회 정기 폴링 | `events:*`, `scheduler:*` | 수동 호출 (또는 외부 cron에서 `aws lambda invoke`) | 낮음 — rule + target 한 줄씩 |

### 상세 설명

**1. Secrets Manager → 환경변수**
- 현재: `CRTFC_KEY=3L6ijr`, `MOCK_MODE=true` 등 Lambda 환경변수
- 전환 방법: SM에 `gbmatch/api-keys` 시크릿 생성 → Lambda 코드에서 `boto3.client('secretsmanager').get_secret_value()` + 전역변수 TTL 캐싱 추가
- 보안 리스크: 환경변수는 Lambda 콘솔에서 평문 노출됨. 데모에서는 문제없지만 실 서비스에서는 SM 필수

**2. KMS → AWS 관리형 키**
- DynamoDB/S3 모두 기본 서버사이드 암호화(SSE) 적용됨
- 차이점: 키 로테이션/삭제 정책을 직접 제어 불가
- 전환 방법: CMK 생성 → 테이블 UpdateTable SSESpecification → S3 BucketEncryption 설정

**3. Bedrock KB → LLM 리랭킹**
- 현재: `search_programs` 2단계에서 Nova Lite가 후보 목록을 적합도순 재정렬
- KB 문서 준비 완료: `s3://gbmatch-documents-973106207635/kb-documents/` (348건 .txt + .metadata.json)
- 전환 방법:
  1. `iam:PassRole` (to bedrock.amazonaws.com) 권한 확보
  2. OpenSearch Serverless 컬렉션 생성 (또는 KB가 자동 프로비저닝)
  3. `bedrock-agent:CreateKnowledgeBase` + `CreateDataSource` 호출
  4. orchestrator의 `_rerank_with_llm()` → `bedrock-agent-runtime:Retrieve` API filter 방식으로 교체
- 메타데이터 구조 (이미 준비됨):
  ```json
  {
    "metadataAttributes": {
      "program_id": {"value": "PBLN_...", "type": "STRING"},
      "area_codes": {"value": ["4700", "1000"], "type": "STRING_LIST"},
      "biz_type_code": {"value": "PC20", "type": "STRING"}
    }
  }
  ```

**4. EventBridge → 수동 호출**
- collector Lambda를 호출하면 수집+만료정리가 동작함
- 전환 방법:
  ```bash
  aws events put-rule --name gbmatch-collector-daily \
    --schedule-expression "cron(0 0 * * ? *)"
  aws events put-targets --rule gbmatch-collector-daily \
    --targets Id=1,Arn=arn:aws:lambda:us-east-1:973106207635:function:gbmatch-collector
  aws lambda add-permission --function-name gbmatch-collector \
    --statement-id EventBridge --action lambda:InvokeFunction \
    --principal events.amazonaws.com
  ```

## B. 권한 없어서 미구현 + 실 서비스에서 반드시 필요한 항목

이 항목들은 우회가 불가능하거나, 우회하면 보안/운영 수준이 떨어지므로 **권한 확보 후 반드시 구현**해야 합니다.

| # | 항목 | 필요 권한 | 이유 | 긴급도 |
|---|---|---|---|---|
| 1 | Bedrock KB 생성 | `iam:PassRole` (to `bedrock.amazonaws.com`) | LLM 리랭킹은 latency/비용/정확도에서 KB에 열위. 350건 이상 시 더 벌어짐 | 🟡 높음 |
| 2 | OpenSearch Serverless 컬렉션 | `aoss:*` | KB의 벡터 저장소. KB 생성 시 자동 프로비저닝되므로 1번과 같이 해결됨 | (1번에 포함) |
| 3 | EventBridge 스케줄 | `events:PutRule`, `events:PutTargets` | 공고 자동 갱신. 없으면 매일 수동 호출 필요 | 🟡 높음 |
| 4 | Secrets Manager 시크릿 생성 | `secretsmanager:CreateSecret`, `secretsmanager:PutSecretValue` | API 키 평문 노출 방지 (1-16). 현재 환경변수에 노출됨 | 🟠 중간 |
| 5 | KMS 고객관리형 키 | `kms:CreateKey`, `kms:CreateAlias` | 암호화 키 라이프사이클 관리. 현재 AWS 관리형으로 동작은 함 | 🟢 낮음 |

### 권한 요청 시 참고

**최소한 1~3번만 확보하면** Phase 1~5가 스펙과 완전히 일치합니다.
