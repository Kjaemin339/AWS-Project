## 1. 프로젝트 개요

경북 지역 제조·전자 중소기업이 흩어져 있는 정부 지원사업 공고를 일일이 찾아보지 않아도, **기업 프로필 기반 자동 매칭(하드필터+RAG) → 지원서 초안 자동 생성 → 기대효과 시각화 → 챗봇 Q&A**까지 한 번에 제공하는 웹 플랫폼.

### 1-1. 배경 및 문제 정의
- 경북 제조·전자 중소기업은 담당자가 1명뿐인 경우도 많아, 수시로 쏟아지는 지원사업 공고를 일일이 찾아보고 자격요건을 대조할 여력이 없다.
- 공고문은 형식이 제각각이고 행정용어 위주라, "우리 회사가 이거 받을 수 있는지"조차 파악하는 데 시간이 오래 걸린다.
- 지원서 작성 자체도 매번 처음부터 다시 써야 하는 반복적인 시간 낭비 작업이다.

### 1-2. 솔루션 요약
1. **매칭 (하드필터 + RAG)**: 기업 프로필(업종, 근로자수 구간, 소재지, 업력 구간, 매출액 구간, 희망 지원분야, 사업자번호)을 입력하면, 업종·자격요건·지역·마감일로 1차 필터링한 뒤 KB 의미검색으로 후보를 순위화하여 Top 3~5를 조건 근거와 함께 제시.
2. **사업자번호 기반 인증서 자동 확인**: 이노비즈/벤처기업/메인비즈 인증 보유 여부를 자동 조회하여 매칭 가산 조건으로 활용.
3. **지원서 초안 생성**: 사업 개요/신청 사유/기대효과 서술을 AI가 자동 생성하고, 수치(금액/마감일 등)와 **온라인 신청 링크·문의처**는 공고 마스터 테이블에서 조회한 값을 그대로 삽입.
4. **기대효과 시각화**: 규칙기반 계산식으로 예상 매출증가/고용창출/예상 순이익/회수기간을 산출하고, LLM이 자연어로 설명, 차트로 시각화.
5. **챗봇 Q&A**: 매칭 결과나 초안에 대해 자유 질의.

### 1-3. 차별점 (경쟁 우위)
1. **"검색"에서 끝나지 않고 "매칭 → 작성 → 시각화 → Q&A"까지 이어지는 원스톱 흐름.**
2. **자격요건을 놓치지 않는 2단계 매칭.** 업종·지역·규모·업력 코드로 먼저 걸러낸 뒤 AI가 순위와 근거를 설명한다.
3. **실제 공식 데이터 기반 RAG.**
4. **인증서 자동 확인.**
5. **작은 회사, 적은 리소스를 위한 설계.**

## 4. 사용자 흐름 (End-to-End)

```
회원가입/로그인 (Cognito)
   ↓
기업 프로필 입력 (업종, 근로자수 구간, 소재지, 업력 구간 *예비창업 체크 시 비활성화*,
                 매출액 구간(선택), 희망 지원분야, 사업자번호(선택))
   ↓
[자동] verify_certifications 호출 (사업자번호 입력 시)
   ↓
[자동] search_programs 호출
   1) 마스터 테이블에서 업종/지역코드/근로자수구간/업력구간/마감일 하드필터
      ('|' 구분 다중값 필드는 분리 후 포함 여부 확인)
   2) 필터 통과 후보를 KB 의미검색으로 순위화 (보유인증·매출액구간 일치 시 가산)
   3) 회사 프로필 + 후보를 LLM에 전달해 매칭 근거 문장 생성 (텍스트 필드 사용, 코드값 노출 금지)
   4) 후보 0건이면 조건 완화 제안 반환
   ↓ (매칭 이력 신규 레코드 생성)
매칭 Top 3~5 표시 (근거 포함)
   ↓
공고 상세 화면 (지원대상/금액/마감일/신청링크/문의처 표시) → "초안 생성" 클릭
   ↓
[자동] generate_draft 호출 → 초안 + 신청 링크 안내 표시
   ↓
[자동] calc_expected_effect 호출 → 매출증가/고용창출/예상순이익/회수기간 표시
   ↓
(선택) 챗봇 질문
   ↓
마이페이지 이력 재조회
```

## 6. 회사 프로필 및 매칭 설계

### 6-1. 회사 프로필 입력 항목 (API 코드체계 정렬)

| 항목 | 사용자 표시 | 내부 코드 | 근거 |
|---|---|---|---|
| 업종 | 제조업 세부분류 (전자부품/기계/자동차부품/금속가공 등, 자유 선택) | 텍스트 (`induty` 매칭용) | hashTags/description에서 추출 |
| 근로자수 | 1~5명 미만 / 5~10명 / 10~20명 / 20~50명 / 50~100명 / 100명 이상 | `EI01`~`EI06` | MANUAL 데이터 매칭용 (공고정보 API 미제공) |
| 소재지 | 경상북도 고정 + 시군(자유 입력) | `areaCd=4700` 또는 `1000` | hashTags에서 지역 추출 |
| 업력 | 3년 미만 / 3~5년 / 5~7년 / 7~10년 / 10~20년 / 20년 이상 | `OI01`~`OI06` | MANUAL 데이터 매칭용 (공고정보 API 미제공). **예비창업 체크 시 비활성화(입력 불가)** |
| 예비창업 여부 | 별도 체크박스 (**업력과 상호배타적**) | `fntnYn` (Y/N) | 체크 시 업력 필드는 자동 숨김/비활성화, 저장 시 업력값 null 처리 |
| 매출액 구간 (v2.3 복원) | 5억 미만 / 5~10억 / 10~20억 / 20~50억 / 50~100억 / 100~300억 / 300억 이상 (선택 입력, 스킵 가능) | `SI01`~`SI07` | MANUAL 데이터 매칭용 (공고정보 API 미제공). 미입력 시 2단계 가산조건에서 제외 |
| 희망 지원분야 | R&D / 시설·장비투자 / 인력양성 / 수출·판로 / 자금·보증 (다중선택) | `sportTypeCd` 매핑 (6-2 참고) | API `lcategory`/`searchLclasId` 분야코드 매핑 |

**참고 — 공고정보 API의 구조화 코드 부재**: 공고정보 API(기업마당)는 근로자수/업력/매출액/인증 관련 구조화 코드를 제공하지 않는다. 따라서 API 수집 공고는 해당 필드가 빈값으로 저장되며, 하드필터에서 해당 조건을 자동 통과한다. 이 코드 체계는 MANUAL(폴백) 데이터에서만 활용된다.

→ "귀사는 경북 구미 소재 업력 10~20년(OI05) 전자부품 제조업으로, R&D 지원을 원하시니 이 공고가 맞습니다" 식의 근거 있는 매칭 가능.

### 6-2. 매칭 로직 — 2단계 (하드필터 + AI 랭킹/설명)

```
1단계 (규칙 기반 하드필터, 공고 마스터 테이블 대상)
  - induty(업종) 일치 여부 (v2.3 복원)
    · 공고의 induty가 "전체/공통" 또는 사용자 업종과 일치(텍스트 키워드 매칭, 완화 적용)
  - area_code에 "4700"(경상북도) 또는 "1000"(전국) 포함 여부
    · API 공고: hashTags에서 추출한 지역 텍스트로 판별
  - employee_count_code 요건 충족 여부
    · API 공고: 빈값(API 미제공)이면 자동 통과. MANUAL 데이터에만 적용
  - business_age_code 요건 충족 여부 (예비창업 대상 공고는 fntnYn 별도 확인)
    · API 공고: 빈값(API 미제공)이면 자동 통과. MANUAL 데이터에만 적용
  - 마감일 deadline >= 오늘
    · API 공고: reqstDt에서 파싱

  ※ MANUAL(폴백) 데이터의 employee_count_code/business_age_code/need_certification_code/
    sales_amount_code는 '|' 구분 다중값이 가능하므로, 하드필터·가산조건 비교 전
    반드시 '|'로 분리한 뒤 각 값을 개별 확인한다. API 수집 공고는 이 필드들이 빈값이므로
    해당 필터를 자동 통과(빈값 = 조건 없음으로 간주)한다.

2단계 (AI 기반 순위화 및 설명)
  - 1단계 통과 후보만 대상으로 KB 의미검색으로 희망 지원분야와의 관련도 순위화
  - 보유 인증(need_certification_code 일치) 및 매출액 구간(sales_amount_code 일치, 입력된 경우만) 시 순위 가산
    — 둘 다 필수 조건이 아닌 가산 조건으로만 사용. API 공고는 해당 필드 빈값이므로 가산 없음
  - 회사 프로필 + 상위 후보 정보를 LLM에 전달해 매칭 근거 문장 생성
    (코드값이 아닌 지역명/업종명/사업유형명 등 텍스트 필드를 사용해 사람이 읽을 수 있는 근거로 작성)

예외 처리
  - 1단계 통과 후보가 0건이면, 제약이 된 필터 항목(업종/지역 중 어떤 것인지)을
    안내하고 조건 완화 후 재검색을 제안한다.
```

## 7. 데이터 소스 확보 전략 (확정)

### 7-1. 확정된 두 공식 API

| API | 제공기관 | URL | 용도 | 인증 방식 | 인증키 상태 |
|---|---|---|---|---|---|
| 공고정보 API | 기업마당 (bizinfo.go.kr) | `https://www.bizinfo.go.kr/uss/rss/bizinfoApi.do` | 지원사업 공고 정보 | query parameter (`crtfcKey=`) | ✅ 발급 완료 |
| 증명(확인)서 일괄확인 API | 중소벤처24 | — | 사업자번호 기반 인증서 확인 | header (`Authorization: Bearer {token}`) 권장 | 승인 대기 |

- **공고정보 API**: 기업마당(bizinfo.go.kr)에서 제공하는 공고정보 RSS/API. 인증키(`crtfcKey=3L6ijr`)는 이미 발급 완료 상태이다. 요청 시 `searchLclasId` 파라미터로 분야별 필터링 가능(01=금융, 02=기술, 03=인력, 04=수출, 05=내수, 06=창업, 07=경영, 09=기타).
- **증명서 API**: 중소벤처24에서 제공. 현재 승인 대기 중이며, 지연 시 TIPA 담당자(윤재철 주임연구원, 044-300-0764)에게 문의한다.

**공고정보 API 주요 응답 필드**: `seq`(공고ID, 예: `PBLN_000000000080236`), `title`, `link`, `author`, `description`/`bsnsSumryCn`(사업요약), `lcategory`, `hashTags`(지역/분야 텍스트), `reqstDt`/`reqstBeginEndDe`(신청기간), `trgetNm`(지원대상), `refrncNm`(문의처), `rceptEngnHmpgUrl`(신청URL). 기존 공고민간연계 API와 달리 근로자수코드/업력코드/매출액코드/인증코드/지원금액 등 구조화 코드는 제공하지 않으므로, 해당 정보는 텍스트(hashTags/description)에서 추출하거나 빈값으로 저장한다.

### 7-2. 크롤링에서 API 연동으로 전환
정기 수집(EventBridge)의 실제 구현은 웹 스크레이핑이 아니라 **공고정보 API(bizinfo.go.kr) 정기 폴링**으로 확정된다. API는 `searchLclasId` 파라미터로 분야별 필터링(01=금융~09=기타)을 제공하지만, 지역/업종 등 세부 조건 필터는 우리 백엔드가 `hashTags`/`description`에서 직접 추출·판별한다.

**데이터량 관리 (v2.3 추가)**: 전국 대상 공고까지 포함하면 예상보다 데이터량이 커질 수 있다. `searchLclasId`로 분야를 한정하고, 최근 N일 공고 위주로만 수집하며, 필요 시 전국 공고는 별도로 등록건수 상한을 두는 것을 검토한다.

### 7-3. 1단계 수동 확보 (폴백)
API 장애/데이터 부족에 대비해 5개 카테고리(R&D/시설장비투자/인력양성/수출판로/자금보증) 공고 원문을 수동 확보한다.

**폴백 데이터의 식별자 채번 규칙 (v2.3 신설, 최종 재확인 시 보완)**: `program_id`는 DynamoDB 파티션 키 타입 제약(테이블당 하나의 타입만 허용)에 따라 **문자열(String) 타입으로 통일**한다. API 수집분은 `seq` 필드(예: `"PBLN_000000000080236"`)를 그대로 문자열로 저장하고, 수동 확보분은 **`program_id = "MANUAL-001"`~`"MANUAL-005"`** 형식의 문자열 식별자를 자체 채번한다. 이렇게 하면 문자열 접두사만으로 두 데이터 출처를 구분할 수 있다.

### 7-4. 문서 포맷 처리 원칙
첨부파일이 `.hwp`이면 PDF로 변환한다. 실제 포맷은 개발 착수 시 샘플 호출로 확인한다.

## 8. 기대효과 계산 — 규칙기반 계산식 + LLM 설명 (하이브리드)

```
기대 매출증가(원)   = 지원금액 × 업종별 평균 매출승수(더미 테이블)
기대 고용창출(명)   = 지원금액 / 업종별 평균 1인당 투자비용(더미 테이블)
예상 순이익(원/년)  = 기대 매출증가 × 이익률 가정치        ← v2.3 신설 (절대값 지표)
회수기간(년)        = 지원금액 / 예상 순이익
```

### 계산식의 수학적 성질 (v2.3 명시)
회수기간 공식을 전개하면 `회수기간 = 지원금액 / (지원금액 × 매출승수 × 이익률) = 1 / (매출승수 × 이익률)`이 되어, **회수기간 자체는 지원금액 규모와 무관하게 업종별 매출승수·이익률에 의해서만 결정되는 비율 지표**가 된다. 이는 이 단순 선형 모델의 자연스러운 수학적 결과이며 버그가 아니지만, 사용자에게는 "왜 지원금액이 다른데 회수기간이 같냐"는 의문을 살 수 있다. 이를 보완하기 위해 **예상 순이익(절대값, 원 단위)** 지표를 별도로 제시하여, 지원금액 규모에 따른 차이는 이 절대값 지표로 확인하도록 안내한다.

### 입력 계약 (Input Contract)
`calc_expected_effect`는 `program_id`를 받아 마스터 테이블에서 지원금액을 조회한다. `min_support_amount`/`max_support_amount`가 있으면 우선 사용(중간값 또는 최대값), 둘 다 비어있으면 `overview`/`support_content` 텍스트에서 LLM으로 보완 추출한다. 공고정보 API 수집 공고는 지원금액 필드가 대부분 LLM 추출값이므로, 추출 실패 시 기대효과 계산을 스킵하고 안내 메시지를 반환한다.

## 9. 데이터 흐름 개요

```
[정기 수집] EventBridge → Lambda 수집기
   → 공고정보 API (bizinfo.go.kr) 호출 (searchLclasId 분야 필터)
   → hashTags에서 지역 추출, 경북/전국 관련 공고 1차 선별

   → 지역·업종은 hashTags/description에서 추출, 근로자수/업력/매출액 코드는 API 미제공(빈값 저장)
   → 공고 마스터 테이블 적재 (program_id=seq) + KB 등록
[폴백] 5개 카테고리 수동 확보 PDF → program_id="MANUAL-001~005"로 채번하여 동일하게 적재
   (폴백 데이터는 근로자수/업력/매출액 코드 포함, '|' 다중값 필드 분리 저장)
   ↓ (공통)
사용자 프로필 제출 (+ 인증서 자동확인)
   → search_programs: 업종/지역/마감일 하드필터 (근로자수/업력/매출액 필드가 빈값인 API 공고는 해당 필터 자동 통과)
      → KB 순위화(인증·매출액 가산) → 매칭 근거 생성
   → generate_draft: 수치+신청링크 조회 후 초안 생성
   → calc_expected_effect: 금액 조회 후 매출증가/고용창출/예상순이익/회수기간 계산
```

| 구조화 코드 부재 | 공고정보 API에 근로자수/업력/매출액 구간 코드 없음 — API 수집 공고는 해당 하드필터 자동 통과, 폴백(MANUAL) 데이터에만 적용 | ✅ 방침 확정 |
| 기업규모코드 처리 | 공고정보 API에서 미제공 — 폴백 데이터의 `cmpScaleCd` 저장은 하되 매칭 미사용 유지 | ✅ 방침 확정 |
| 첨부파일 포맷 | hwp/pdf 실제 확인 필요 | 확인 중 |
| 전국 공고 데이터량 | 필요 시 `searchLclasId` 분야 필터 및 최근 N일 제한으로 조절 | 방침 수립됨 |
| 기대효과 계산식 성질 | 회수기간이 지원금액과 무관해지는 수학적 특성 인지, 예상순이익(절대값) 지표로 보완 | ✅ 방침 확정 |
| 초안 정확도 | AI 서술 품질 검증 방법 필요 | 설계 중 |
| 개인정보(사업자등록번호) | 암호화 저장 및 이용약관 명시 필요 | 설계 중 |
| 비용 | KB 벡터스토어 운영 비용 최적화 | 검토 중 |
| VPC 격리 | 규정상 필수 여부 확인 | 확인 중 |
