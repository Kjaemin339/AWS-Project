### 실제 진행 결과

### 계획 대비 변경/보완 사항

1. **KMS 키 미생성**: 랩 계정 정책에 `kms:*` 액션이 포함되어 있지 않아 고객관리형 키를 생성할 수 없음. 암호화가 필요한 경우 AWS 관리형 키(aws/dynamodb, aws/s3 등)가 기본 적용되므로 MVP에서는 이것으로 대체.

2. **Secrets Manager 미사용 → Lambda 환경변수로 대체**: `secretsmanager:*` 권한이 없어 시크릿 생성 불가. API 토큰은 Lambda 환경변수로 관리하되, 실 서비스 전환 시 Secrets Manager + TTL 캐싱(스펙 1-16)으로 교체 필요. 로그 마스킹 원칙은 코드 레벨에서 동일하게 적용.

3. **IAM 역할 생성 시 permissions boundary 필수**: 랩 환경의 `CreateRoleOnlyWithBoundary` 정책에 의해 `agent-lab-group-role-boundary`를 반드시 부착해야 역할 생성 가능. 이 boundary 안에 kms/secretsmanager 액션이 없으므로 Lambda 역할도 해당 서비스 접근 불가.

4. **세 번째 역할명 변경**: 스펙상 명시 없던 세 번째 역할을 `gbmatch-reserve-role`로 명명 (Phase 7 edge는 팀원 담당이므로 용도 미정 예비 역할).

### 계획 대비 변경/보완 사항

1. **API 소스 전면 변경**: 중소벤처24 → 기업마당(bizinfo.go.kr). 인증키 `3L6ijr` 직접 사용 (승인 완료).

2. **구조화 코드 부재 → 빈값 저장 + 하드필터 자동통과**: 기업마당 API는 `employee_count_code`, `business_age_code`, `sales_amount_code` 등을 제공하지 않음. 빈값("")으로 저장하고, Phase 4의 하드필터에서 빈값=자동통과 로직으로 처리.

3. **JSON 응답 구조 차이**: 문서에는 `jsonArray.item[]`으로 명시되어 있으나, 실제 응답은 `jsonArray` 자체가 리스트(`[{...},{...}]`). 양쪽 모두 호환하도록 구현.

4. **area_code 중복 제거**: `dict.fromkeys()`로 같은 코드가 반복 등장하는 것 방지 (예: "경북"+"경상북도" 둘 다 4700 → 1개만).

5. **실행시간 26초**: 8개 카테고리 × 50건 = 최대 400건 수집, DynamoDB put_item 300+ 건. Lambda 타임아웃 60초 내 충분.

6. **Bedrock KB 및 EventBridge는 별도 진행**: KB는 CLAUDE.md 규칙 4(3단계 확인)에 따라 메타데이터 사이드카 구조 확인→KB 생성→filter 테스트 순으로 진행. EventBridge는 1일 1회 cron 설정만 남음.

## Phase 4/6 — 초안 프롬프트 보강 (2026-07-08)

### 문제

### 수정

| 항목 | 내용 |

| 프로필 폼 | 회사명(신규, 필수 아님) + 매출액구간(`SALES_AMOUNT_CODES` SI01~SI07, 선택) 필드 추가. `toApiProfile()`에 `company_name`/`location_detail`/`sales_amount_code` 포함 |

## Phase 4 — 지원금액 "0원" 오표시 버그 수정 (2026-07-08)

### 증상

매칭 결과 카드·공고 상세의 지원금액 배지가 실제로는 값이 없는 공고인데도 "0원"으로 표시됨 (기업마당/bizinfo.go.kr API 수집분 대부분이 `max_support_amount` 구조화 필드를 제공하지 않아 마스터 테이블에 0으로 저장돼 있음 — 이 자체는 기존에 알려진 데이터 한계).

### 근본 원인 (데이터 한계가 아니라 별도의 실제 버그였음)

1. `search_programs`가 반환하는 매칭 결과 요약에 `support_scale`(지원규모 원문 텍스트, 값이 있을 때가 많음) 필드 자체가 누락되어 있어 폴백을 걸 수 없었음
2. **더 근본적인 버그**: DynamoDB에서 읽은 숫자는 boto3가 `Decimal`로 반환하는데, Lambda 응답 직렬화(`json.dumps(..., default=str)`)가 `Decimal(0)`을 **문자열 `"0"`으로 직렬화**함 → JS에서 문자열 `"0"`은 truthy라 `if (program.max_support_amount)` 같은 falsy 체크가 전부 무력화됨. 즉 값이 없어도 "있음" 취급되어 `formatWon("0")` → "0원"이 찍힘

### 수정

| 항목 | 내용 |
|---|---|
| `search_programs` | 매칭 결과 요약에 `support_scale` 추가, `max_support_amount`를 `int(... or 0)`으로 명시 변환해 반환 |
| `get_matched_programs` | 신규 헬퍼 `_decimals_to_native()` 추가 — DynamoDB Item 전체를 재귀적으로 순회해 `Decimal`을 `int`/`float`로 변환 후 응답 (마이페이지 이력 재현 시에도 동일 버그 방지) |
| 프론트 `formatSupportAmount(program)` (mockData.js 신규) | `support_scale` → `max_support_amount` → 그마저 없으면 "0원" 대신 **"지원금액 정보 없음"**을 표시하도록 우선순위 폴백 구성. `Results.jsx`/`ProgramDetail.jsx` 적용 |

### 검증

- curl로 API Gateway 원본 응답 확인: 수정 전 `"max_support_amount": "0"`(문자열) → 수정 후 `"max_support_amount": 0`(숫자)
- 브라우저 E2E: 지원금액 미제공 공고 5건 모두 "0원" → "지원금액 정보 없음"으로 정상 표시 확인
- 참고: 이 Decimal 직렬화 버그는 다른 응답 경로에도 잠재해 있을 수 있어, 앞으로 DynamoDB에서 읽은 값을 그대로 반환하는 곳이 추가되면 `_decimals_to_native()` 적용 여부를 확인할 것

## Phase 4 — LLM 모델 교체: Nova → Claude Haiku 4.5 (2026-07-08)

### 배경

Nova Lite(의도 라우팅·분류)/Nova Pro(초안 서술·설명 생성)로는 서술 품질에 한계가 있어, 더 똑똑한 모델로 교체 요청. 이 계정에서 `aws bedrock list-foundation-models`로 확인한 결과 Claude Haiku 4.5/Sonnet 4.5·5/Opus 4.5~4.8 등 Anthropic 모델 접근 권한이 있었음. 다만 on-demand 직접 호출(`InvokeModel`)은 막혀 있고, cross-region inference profile(`us.anthropic.claude-haiku-4-5-20251001-v1:0`) 경유로만 호출 가능함을 확인.

### 수정

| 항목 | 내용 |
|---|---|
| `invoke_nova`/`invoke_nova_with_guardrail` → `invoke_llm`/`invoke_llm_with_guardrail` | `bedrock.invoke_model()`(Nova 전용 요청/응답 포맷) 대신 **Bedrock Converse API**(`bedrock.converse()`)로 전환 — 모델 무관 통일 인터페이스라 Nova/Claude 어느 쪽으로도 전환 가능. Guardrail은 `guardrailConfig` 파라미터로 그대로 지원 |
| 모델 ID | 전체 LLM 호출(의도 라우팅 포함)을 `LLM_MODEL_ID` 환경변수 하나로 통일, 기본값 `us.anthropic.claude-haiku-4-5-20251001-v1:0` |
| IAM (`gbmatch-orchestrator-policy`) | `BedrockInvoke` Statement의 `Resource`에 Claude inference profile ARN + foundation-model ARN 추가 (기존 Nova ARN은 존치) |
| 프롬프트 6곳 | Claude가 마크다운(`#`, `**` 등)을 즐겨 쓰는데 화면은 plain-text(`monospace`/`pre-line`)라 기호가 그대로 노출되는 부작용 발견 → `_generate_match_rationale`/`generate_draft`/`calc_expected_effect`/`explain_program`/`revise_draft`/`_general_chat_answer` 프롬프트 끝에 "마크다운 문법 쓰지 말고 순수 텍스트로 작성" 지시 추가 |

### 검증

- 직접 invoke: 초안 서술이 Nova 대비 훨씬 자연스럽고 기업 정보(회사명·소재지·근로자수·업력·매출액·인증)를 구체적으로 인용, 마크다운 지시 추가 후 `#`/`**` 없이 순수 텍스트로 출력됨 확인
- API Gateway 경유(`/chat` → `explain_program`)도 정상 동작
- CloudWatch 로그에 Guardrail 관련 예외/폴백 없음 (Converse API + Claude 조합에서도 Guardrail 정상 적용)
- 브라우저 E2E: 초안 생성 화면에서 마크다운 기호 없이 정상 렌더링 확인

### 알려진 한계

- 모델 교체로 토큰당 비용·응답 지연이 Nova 대비 증가할 수 있음 (Haiku 4.5는 Nova Pro보다 품질은 높지만 가격 구조가 다름 — 실사용 트래픽에서 비용 모니터링 필요)
- inference profile 방식이라 IAM Resource ARN에 리전 와일드카드(`arn:aws:bedrock:*::foundation-model/...`)를 사용함 — 더 엄격한 최소권한이 필요하면 실제 라우팅 리전을 특정해 좁힐 것
